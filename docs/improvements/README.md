## How to Read These Documents

Each improvement document follows this structure:

1. **Problem Statement** — What's wrong, with production data as evidence
2. **Root Causes** — Technical analysis of why it happens
3. **Proposed Changes** — Phased implementation plan with specific code changes
4. **Success Criteria** — Measurable targets to validate the fix
5. **Files Affected** — Quick reference for code review
